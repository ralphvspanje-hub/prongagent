## 2026-03-16 — Channel-agnostic messaging

**What happened:** ProngAgent hardcoded "Discord" throughout skill files, workspace files, README, and product docs, but OpenClaw supports 24+ messaging channels (WhatsApp, Telegram, Slack, Signal, etc.) natively via its Gateway layer. The agent never sends to a channel directly — it composes messages and the Gateway routes them. Made the entire repo channel-agnostic so the product accurately reflects the architecture.
**Changed:**
1. **Skill files** — Removed all prescriptive Discord references: `daily-plan/SKILL.md` (description, daily_message mode Steps 5-6), `weekly-review/SKILL.md` (Step 4, examples, tone table, edge cases), `spaced-repetition/SKILL.md` (queue management references), `onboarding/SKILL.md` (post-onboarding step). All now say "message" or "send the message" instead of "Discord message" or "send via Discord". No skill logic, triggers, or evaluation criteria changed.
2. **Workspace files** — `AGENTS.md`: "Discord message formatting" → "Message formatting". `TOOLS.md`: "Discord (built-in)" section rewritten as "Messaging (built-in)" listing all supported channels. `BOOTSTRAP.md`: "if Discord is configured" → "if a messaging channel is configured".
3. **README.md** — Restructured: "Set up Discord" → "Set up a messaging channel" with subsections for Discord, WhatsApp, and Telegram setup commands. "Without Discord" → "Without a messaging channel". Intro updated to list multiple channels. Cron job examples made generic (removed `--channel discord`).
4. **Product docs** — `AGENT_LEARNING_COMPANION.md`: ~30 prescriptive references changed to "messaging". Left comparative table (Section 2 ProngGSD vs Agent) as-is. Architecture diagram says "Messaging (any channel)". `IMPLEMENTATION_GUIDE.md`: ~14 prescriptive references changed. Left historical notes and community references.
5. **CLAUDE.md** — "Daily task generation + Discord delivery" → "+ delivery". "OpenClaw agent + Discord" → "+ messaging" in related projects.
**File:** All skill files, workspace files, README.md, product docs, CLAUDE.md

## 2026-03-16 — Agent self-observation system created

**What happened:** Created a system for the OpenClaw agent to flag issues, surprises, and suggestions about its own skill files without editing them. The developer reviews these observations and decides what to change. Three components: (1) `memory/agent-observations.md` — append-only memory file where the agent logs entries when something about its instructions feels off, unclear, or worth revisiting. Entry format includes: skill file, what happened, what felt off, what the agent did instead, suggested change, severity. (2) Self-observation trigger sections added to all 12 skill files — each skill has both general triggers (uncovered edge case, judgment call not covered by rules, rule that produced wrong result, cross-skill contradiction) and skill-specific triggers tailored to that skill's unique failure modes. (3) Integration with adaptation and weekly-review — adaptation reads observations to check if flagged issues affect its decisions and writes observations when it notices issues during its own trigger analysis; weekly-review reads observations to mention the count in the weekly digest ("I logged N observations this week about my own instructions").
**Changed:** New file `memory/agent-observations.md`. New "Self-observation triggers" section added to: `onboarding.md` (3 triggers: conversation length, unmappable dream career, resume parsing gaps), `daily-plan.md` (3: resource exhaustion, time flexibility usage, ramp-up calibration), `check-in.md` (3: unparseable response, off-plan completions, unfair streak reset), `teach-back.md` (3: question-resource mismatch, gate difficulty, ambiguous response), `adaptation.md` (4: contradicting triggers, cooldown blocking, max changes insufficient, conservative default discomfort), `resource-feedback.md` (2: graduation threshold, rating-behavior mismatch), `spaced-repetition.md` (3: interval calibration, queue growth, bouncing concepts), `weekly-review.md` (2: tone mismatch, misleading comparison), `win-log.md` (3: STAR format forcing, uncovered categories, formulaic extraction), `interview-prep.md` (3: crash course gaps, threshold timing, jarring transition), `mock-interview.md` (3: difficulty miscalibration, contradicting coaching, false pattern detection), `portfolio-projects.md` (3: scope mismatch, irrelevant-despite-matching-pillars, user idea superiority). ## 2026-03-16 — Portfolio projects: dream-career-aligned project suggestion skill created

**What happened:** Phase 3 implementation — wrote the portfolio-projects skill file (the project suggestion engine). Spec from product report Section 3.7 and implementation guide Step 3.3. Four activation paths: skill block completion (pillar levels up — suggest building with the newly leveled skill), building gap (adaptation flags 3+ weeks without Build/Create tasks — nudge toward hands-on work), interview prep (active interview — tailor project to target company's domain), and manual (user asks "what should I build?"). Key design decisions: (1) 5-factor project generation: dream career alignment (what impresses hiring managers in that field), current skill level (one level above comfort zone — Level 1 users get guided tutorial+customize projects, Level 3+ get open-ended builds), skills recently practiced (reinforce don't distract — check history.md for recent pillars), time commitment (relaxed=4-8hr, moderate=8-16hr, intensive=16-30hr), public data/APIs (free resources, no setup barriers), (2) deduplication checks against resume-context.md (don't suggest what they've already built), win-log/wins.md (don't duplicate existing portfolio stories), and interview-mapping.md (fill question type gaps), (3) never forced — decline tracking with frequency reduction after 2 declines (cap at 1/month, only via level-up or manual), reset on manual request, (4) project tasks use standard daily-plan format with action verbs Design/Build/Implement/Test/Deploy/Document, Type=practice, URL=N/A, mixed into regular weekly tasks (1-2 project tasks per day alongside learning tasks), (5) completion triggers win-log STAR draft + mock interview offer if interview prep is active, (6) stalled project check-in after 2+ weeks of skipped tasks — offer smaller scope, pause, or drop without guilt, (7) beginner-specific handling: Level 1 users get guided projects (tutorial + customize), (8) pre-check gates: skip suggestion if completion <70%, skip if interview <14 days away, skip if 3+ active projects exist, (9) interview-prep-specific: smaller scope (4-6hr), company domain match, no stretch goal, front-loaded completion.
**Changed:** New skill file with: 4 trigger paths (skill block completion, building gap, interview prep, manual) each with detection logic and pre-checks, 10-file read scope, 2-file write scope + 2 downstream skill triggers (win-log on completion, mock-interview if interview active), 5-factor project generation system with skill level calibration table, project suggestion format with emoji selection guide, 3-step acceptance flow (add to current-plan → break into 4-8 tasks → mix into weekly plan) with task progression template, 4-step completion flow (update status → trigger win-log → celebrate → interview prep hook), tone rules with verbosity table, 4 conversation examples (skill block completion, interview prep, building gap, user's own idea), 12 edge cases (decline handling, user's own idea, overwhelmed, abandoned projects, complete beginners, interview <2 weeks, 3+ active projects, paid tools, simultaneous level-ups, consumption-heavy learners, completion + level-up coincidence, tight interview timeline).

## 2026-03-16 — Spaced repetition: review scheduling + retention skill created

**What happened:** Phase 2 implementation — wrote the spaced-repetition skill file (the retention engine). Two activation paths: queue management (daily, before daily message — picks the most urgent due concept and writes a review question to the Review Queue for daily-plan to embed) and response processing (after user answers — binary correct/wrong evaluation, updates intervals per the simplified spacing algorithm from the product report). Key design decisions: (1) binary evaluation only (correct/wrong), not the 4-tier teach-back scale — SRS is a quick check, not a deep interaction, (2) priority selection when multiple items are due: overdue first → lowest consecutive correct → dream career relevance, (3) teach-back always takes precedence over SRS if both target the same concept on the same day, (4) concept lifecycle: entry from teach-back (interval based on response quality) or completed blocks (3-day initial) → active review → retirement at 5+ consecutive correct with 3+ month interval, (5) interview prep activation pulls relevant Active AND Retired items to due-today as a one-time burst, (6) skip detection: 5 consecutive unanswered reviews across any concepts triggers frequency reduction to 1/week, (7) question formulation varies format on repeat reviews and calibrates to pillar level, (8) concepts from dropped pillars keep reviewing — foundational knowledge doesn't decay by plan change.
**Changed:** New skill file with: two trigger paths (queue management + response processing), 5-file read scope, 2-file write scope, 5-step queue management flow (read → prioritize → conflict check → formulate → write), 6-step response processing flow (identify → evaluate → apply spacing algorithm → update table → acknowledge → cross-reference), concept lifecycle (entry → active → retiring → retired), spacing algorithm table matching product report Section 3.10, question formulation rules with level calibration table, 3 worked examples (queue management, correct answer, wrong answer), 9 edge cases.

## 2026-03-16 — Daily-plan: flexible task count + time flexibility

**What happened:** CTO review identified two rigidities: (1) the task count table was treated as a rule, forcing 2-3 tasks for 1hr users even when a single deep-focus session would be better, (2) the time commitment was a hard ceiling, so a perfect 1.5hr resource could never be assigned to a 1hr user. Both hurt learning quality.
**Changed:**
1. Task count table is now a guideline, not a rule. Added explicit "single-task days are fine" section with criteria: resource is longer but well-matched, task is practice-heavy benefiting from sustained focus, or user feedback shows preference for deep-dive formats.
2. Added time flexibility rule: daily commitment is a target, not a hard cap. Exceptional resources can exceed by up to 50%. When exceeding: assign only that one task, explain in daily message, never exceed consecutive days.
3. Pillar distribution updated: single-task days should rotate pillars.
4. Type mix reframed: balance over a week, not per day. Full practice day + full conceptual day is fine.
**File:** `skills/daily-plan/SKILL.md`
